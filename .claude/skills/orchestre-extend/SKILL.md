# Skill: orchestre-extend

## Metadata
- **Name**: orchestre-extend
- **Description**: Adds a feature to an existing project while respecting its architecture. Works even on projects not generated by Orchestre.
- **Trigger**: /orchestre-extend, "add a feature", "ajoute une feature", "extend this project"

## Parameters
- `prompt` (required): Description of the feature to add
- `--stack` (optional): Force stack detection (nextjs-supabase | sveltekit-drizzle)

## Process

### STEP 1: Architecture Scan (30 seconds)

Before writing any code, scan the existing project to understand its architecture.

**Read these files (if they exist):**
- `package.json` or `Cargo.toml` or `pyproject.toml` — detect framework/language
- `tsconfig.json` — detect import aliases
- Folder structure: `ls src/` or `ls app/` — detect routing pattern
- `CLAUDE.md` or `PROJECT.md` — detect if Orchestre-generated
- `.orchestre/intent.json` — detect existing feature map

**Detect and report:**
```
--- ARCHITECTURE SCAN ---
Framework: {Next.js / SvelteKit / FastAPI / unknown}
Language: {TypeScript / Python / etc.}
Import alias: {@ / $lib / none}
Routing: {app/ router / file-based / etc.}
Business logic: {lib/ / src/lib/server/ / app/services/ / mixed}
Auth: {Supabase / Lucia / NextAuth / custom / none}
DB: {Supabase / Drizzle / Prisma / SQLAlchemy / raw}
Orchestre-managed: {yes / no}
Architecture score: {clean / acceptable / chaotic}

Existing features detected:
  - {feature 1} ({files})
  - {feature 2} ({files})
---
```

If `Architecture score: chaotic`, suggest `/orchestre-recover` instead:
```
This project's architecture is chaotic. Adding features will make it worse.
Recommendation: run /orchestre-recover first to restructure, then /orchestre-extend.
Continue anyway? (y/n)
```

### STEP 2: Feature Planning (1 minute)

Based on the scan + the user's description, plan the feature:

```
--- FEATURE PLAN ---
Feature: {name}
Description: {1-2 sentences}

Files to CREATE:
  - {path} — {purpose}
  - {path} — {purpose}

Files to MODIFY:
  - {path} — {what changes and why}

Dependencies:
  - {npm package or DB table if needed}

Architecture alignment:
  - Business logic → {where it goes}
  - UI components → {where they go}
  - Schema/types → {where they go}
  - Route/page → {where it goes}
---
```

Show this plan and ask for confirmation via AskUserQuestion:
```
Implement this feature plan?
  Y = Go
  edit = Modify the plan
  n = Cancel
```

### STEP 3: Implementation

Generate the feature following the detected architecture:

1. **Schema first** — Create Zod/Pydantic schema if the feature has data
2. **Business logic** — Queries/mutations in the logic layer (lib/, src/lib/server/, etc.)
3. **UI components** — Pure UI, data via props
4. **Route/page** — Wire everything together
5. **Update existing files** — Navigation, types, imports

**Critical rules:**
- Match the existing import style (alias, barrel exports)
- Match the existing error handling pattern (Result<T> if present, try/catch if not)
- Match the existing naming conventions (camelCase, kebab-case, etc.)
- Never refactor existing code unless necessary for the feature
- Never change existing tests unless the feature breaks them

### STEP 4: Verification

After implementation:
1. Check for TypeScript errors: `npx tsc --noEmit` (if TypeScript project)
2. Check for import errors: verify all imports resolve
3. Run existing tests if they exist: `npm test` (don't fail if no tests)

Report:
```
--- FEATURE ADDED ---
Feature: {name}
Files created: {N}
Files modified: {N}

Verify:
  1. {specific thing to check manually}
  2. {specific thing to check manually}

Next: npm run dev and test the feature
---
```

## Rules

1. **NEVER refactor the whole project** — only touch what's needed for the feature
2. **ALWAYS scan before writing** — understand the existing architecture first
3. **MATCH existing patterns** — don't impose Orchestre conventions on non-Orchestre projects
4. **On Orchestre projects** — follow R1-R8 strictly, use the stack's conventions
5. **CREATE, don't rewrite** — prefer new files over modifying complex existing ones
6. **If uncertain about architecture** — ask the user via AskUserQuestion
